정보보안기사 2장 - (3) 비대칭키 암호

2019-11-30
#정보보안기사#보안

1. 비대칭키 암호

(1) 키 배송 문제

1) 개요

  1. 대칭키 암호를 사용하려고 하면 키 배송 문제(Key Distribution Problem) 발생
  2. 키를 보내지 않으면 수진자 Bob은 수신한 암호문 복호화 불가
    키를 보내면 도청자 Eve도 복호화 가능
    이것이 대칭키 암호의 키 배송 문제
  3. 키 배송 문제 해결 방법
  • 키의 사전 공유에 의한 해결
  • 키배포 센터에 의한 해결(온라인 키 분배)
  • Diffie-Hellman 키 교환에 의한 해결
  • 공개키 암호에 의한 해결

2) 키의 사전 공유에 의한 해결

  1. 키 사전 분배는 키 관리기관(TA, trusted authority)이 사전에 임의의 두 사용자 (A, B)에게 비밀 경로를 통하여 임의 키 KA,B=KB,AK_{A,B} = K_{B,A}KA,B​=KB,A​를 선택하여 전달하는 방법
  2. 일반적으로 TA와 네트워크 상 모든 사용자 사이에 안전한 통로 필요
    사용자가 많은 경우 TA는 물론 사용자들도 많은 키를 관리해야 하는 문제점 존재
  3. nnn 명의 사용자 있을 시 각 사용자는 n−1n-1n−1가지 키 관리해야 하며 TA는 n(n−1)2{n(n-1) \over 2}2n(n−1)​가지 키 관리해야 함
    매우 복잡하고 관리 비용 많이 듬
  • 키 관리 기관(KDC, Key Distribution Center)과 키 관리기관(TA, trusted authority) 같다고 봐도 무방

3) 키배포 센터에 의한 해결(온라인 키 분배)

(가) 개요

  1. 암호 통신이 필요해질 때마다 통신용 키를 키배포 센터(KDC, Key Distribution Center)라는 신뢰받는 제3자에 의로하여 개인과 키배포 센터 사이에서만 키를 사전에 공유하는 방법
    (TA가 네트워크 상 모든 사용자와 필요할 때마다 키를 공유하는 것과 같음)
  2. 회사 안에 키배포 센터 역할의 컴퓨터를 지정, nnn명의 사원 있다면 nnn개의 키를 컴퓨터 데이터베이스에 저장.

(나) 단계별 순서

  1. Alice는 키배포 센터에 Bob과 통신 요청
  2. 키배포 센터는 의사난수 생성기 사용해 세션키(K) 만듬.
    A와 B의 이번 통신만을 위한 일시적인 키
  3. 키배포 센터는 데이터베이스에서 A의 키(KA)와 B의 키(KB)를 꺼냄
  4. 키배포 센터는 A의 키 사용하여 세션키를 암호화(CA=EKA(K)C_A = E_{KA}(K)CA​=EKA​(K))해서 A에게 보냄
  5. 키배포 센터는 B의 키 사용하여 세션키를 암호화(CB=EKB(K)C_B = E_{KB}(K)CB​=EKB​(K))해서 B에게 보냄
  6. A는 키배포 센터에서 온 세션키(KA로 암호화 됨)를 복호화(K=DKA(CA)K = D_{KA}(C_A)K=DKA​(CA​))해서 세션키 얻음
  7. B는 키배포 센터에서 온 세션키(KB로 암호화 됨)를 복호화(K=DKB(CB)K = D_{KB}(C_B)K=DKB​(CB​))해서 세션키 얻음
  8. B는 세션키 사용하여 A에게서 온 암호문을 복호화(M=DK(C))M = D_K(C))M=DK​(C)))함
  9. A와 B는 세션키를 삭제한다.

4) Diffie-Hellman 키 교환에 의한 해결

(가) 개요

  1. Diffie-Hellman 키 교환(Diffie-Hellman key exchange)은 최초로 공개키 암호방식을 제안한 휘트필드 디피(Whitfield Diffie)와 마틴 헬먼(Martin Hellman)이 발명한 알고리즘
  2. 공개키 암호방식의 개념을 이용하여 두 사용자 간 공통의 암호화키를 안전하게 공유할 수 있는 법을 제시하였고, 많은 키 분배 방시에 관한 연구의 기본이 됨
    (최초의 비밀키 교환 프로토콜)
  3. Diffie-Hellman 프로토콜 방법에서는 양쪽 통신주체가 KDC 없이 대칭 세션키를 생성함
    대칭키를 만들기 전에 양쪽은 두 개의 수 ppp와 ggg를 선택해야 한다.
    여기서 ppp는 매우 큰 소수로서 300자리가 넘는 10진수(1024비트)이다.
  4. 키 교환이라는 이름을 갖고 있지만 실제로는 키 교환이 아닌 공유할 키를 계산하여 만들어 내는 꼴
    때문에 Diffie-Hellman 키 합의(Diffie-Hellman key agreement)라 불리기도 함
  5. 유한체상의 이산대수 문제(DLP, Discrete Logarithm Problem)를 풀기 어렵다는 사실이 Diffie-Hellman 키교환을 뒷받침
    이산대수 문제 : GAG^AGAmodPPP에서 수 A를 구하는 문제
  • ppp와 ggg는 비밀이여야 할 필요 없고 도청자에게 알려줘도 문제 없음
  • 타원 곡선 Diffie-Hellman 키 교환(ECDH) : 이산대수 문제를 타원곡선상의 이산대수 문제로 대체한 키 교환 알고리즘도 있음
    Diffie-Hellman 키 교환보다 짧은 키 길이로 높은 보안성 실현 가능

(나) 절차

  1. Alice는 임의의 큰 수 xxx를 0≤x≤p−10 \leq x \leq p-10≤x≤p−1 안에서 선택하고 R1=gxR_1 = g^xR1​=gxmodppp를 계산
  2. Bob은 다른 임의의 큰 수 yyy를 0≤y≤p−10 \leq y \leq p-10≤y≤p−1 안에서 선택하고 R2=gyR2 = g^yR2=gymodppp를 계산
  3. A는 R1R_1R1​을 B에게 보냄. 여기서 A는 x의 값이 아닌 오직 R1R_1R1​만 보냄.
  4. B는 R2R_2R2​을 A에게 보냄. 여기서 B는 y의 값이 아닌 오직 R2R_2R2​만 보냄.
  5. A는 K=(R2)xK = (R_2)^xK=(R2​)xmodppp를 계산
  6. B는 K=(R1)yK = (R_1)^yK=(R1​)ymodppp를 계산
  7. 이렇게 구한 값 K는 아래와 같고 이것이 세션에 사용될 대칭키임
    K=(gxK = (g^xK=(gx mod p)yp)^yp)y mod p=(gyp = (g^yp=(gy mod p)xp)^xp)x mod p=gxyp = g^{xy}p=gxy mod ppp

  • 모듈러 연산 성질

    1. (a+b) mod n = ((a mod n) + (b mod n)) mod n
    2. (a-b) mod n = ((a mod n) - (b mod n)) mod n
    3. (axb) mod n = ((a mod n) x (b mod n)) mod n

(다) Diffie-Hellman의 안정성

  1. 이산대수 공격

    • 키 교환의 안정성은 이산대수 문제를 풀기 어렵다는데 기반을 두고 있음
    • 도청자 Eve가 R1R_1R1​, R2R_2R2​ 가로챌 수 있음. 만약 E가 R1=gxR_1 = g^xR1​=gx mod ppp에서 xxx를 구하고 R2=gyR_2 = g^yR2​=gy mod ppp에서 yyy를 구할 수 있다면 E는 대칭키 K=gxyK = g^{xy}K=gxy mod ppp 계산 가능. 이렇게 되면 더 이상 비밀이 되지 못함.

  2. 서비스 거부 공격(DoS, Denial of Service)

    • DH 기법은 지수 함수에 기초하고 있으므로 계산이 복잡하여 많은 수의 장치와 동시에 통신을 하는 경우 비밀키 생성에 큰 지연 시간 발생 가능
    • 따라서 DH 기법은 제3자에 의한 DoS 공격에 대한 취약성을 가지고 있음. 악의적인 제3자가 공격 대상 서버에 대해 IP 스푸핑 등을 통해 위조한 키 생성 요청을 동시에 다수 요청함으로써 키 생성 부담으로 서버가 마비되도록 공격 가능.

  3. 중간자 공격(man-in-the-middle attack)

    • 키 교환 프로토콜은 이런 공격에 취약함
      인증단계가 없기 때문
      이런 공격 막기 위해서는 전자서명과 공개키 인증서 등 이용하면 됨
  • 중간자 공격 : 양동이 소방대 공격(bucket brigade attack)이라고도 함
    마치 불을 끄려고 한줄로 늘어선 자원 봉사자들이 물이든 양동이를 옆사람에게 전달하는 모양처럼 보이기 때문 Diffie-Hellman에 기반을 둔 방법에서는 이런 공격 막기 위해 인증 사용, 이를 방어하려고 TLS 통신 생김
  • 국-대-국 키 합의 : 국-대-국 프로토콜(STS, Station-To-Station protocol)은 Diffie-Hellman에 기반을 둔 방법
    여기서 Alice, Bob 사이의 세션키 만들기 위해 공개키 인증서 이용한 전자서명 사용
    국-대-국 프로토콜은 중간자 공격에 안전

(라) Diffie-Hellman 알고리즘의 응용

  1. 중간자 공격의 취약성 문제는 세션키를 생성하는 통신 당사자 A와 B가 교환하는 메시지에 대한 인증을 통해 해결 가능
    IP 스푸핑을 통한 DoS 공격은 키 생성 요청자를 확인하기 위한 쿠키(cookie)를 사용함으로써 방지 가능
  2. 현재 DH 기법 기반의 대칭키 분배는 중간자 공격과 DoS 공격 취약성을 보완하여 IPSec, SSL 등 다양한 인터넷 보안 프로토콜에 광범위하게 적용됨

5) 공개키 암호에 의한 해결

  1. 대칭키 암호에서 암호화키와 복호화키는 같음
    그러나 공개키 암호에서는 암호화키와 복호화키 다름
  2. 수신자는 미리 암호화키를 송신자에게 알려줌 이 암호화키는 도청자에게 알려져도 괜찮음
    송신자는 암호화키로 암호화하여 수신자에게 보냄
  3. 복호화 할 수 있는 것은 복호화키 가지고 있는 수신자뿐
    복호화키는 수신자에게 배송될 필요 없음

(2) 공개키 암호(public-key cryptography)

1) 개요

  1. 대칭키 암호는 평문을 복잡한 형태로 변환하여 기밀성 유지
    공개키 암호는 수학적으로 해결하기 곤란한 문제를 토대로 기밀성 유지
  2. 공개키 암호에서는 암호화키와 복호화키 분리되어 있음
    송신자는 암호화키 사용하여 메시지 암호화 하고, 수신자는 복호화키 사용 암호문 복호화 함
  3. 키쌍을 이루고 있는 2개의 키는 서로 밀접한 관계(수학적 관계) 있음
    이 때문에 공개키와 개인키를 각각 별개로 만들 수 없음

  • 공개키 사용 원칙

    • 암호화키와 복호화키는 같은 사람의 키쌍이어야 제시
    • 키는 암 * 복호화 중 한 번만 사용해야 함
    • 타인의 개인키 사용 불가해야 함

  • 공개키 암호화
    소인수분해 문제(Integer Fatorization problem)

    • RSA
    • Rabin

    이산대수 문제(Discrete Logarithms Problem)

    • ElGamal
    • DSA
    • ECC(타원곡선)

2) 공개키 암호의 역사

  1. 디피, 헬만이 공개키 암호화 아이디어 최초 발표
  2. 당시 구체적 알고리즘 없이 어떤 특성 갖춰야 하는지 제시
  3. Ron Rivest, Adi Shamir, Leonard Adleman이 RSA 발표, 공개키 암호의 De Facto Standard(사실 표준)이라 볼 수 있음

3) 대칭키 vs 비대칭키

  1. 비대칭키(공개키) 암호시스템의 등장으로 인해 대칭키(비밀키) 암호시스템 역할 없어지지 않음
  2. 비대칭키 암호시스템이 암*복호화에 수학적 함수 사용하여 속도가 대칭키 암호시스템보다 무척 느리기 때문
    메시지 크기 큰 경우 대칭키 암호시스템 역할 중요
  3. 비대칭키 암호시스템은 인증, 전자서명, 대칭키 교환에 유용하게 사용됨

  • 공개키 암호에 대한 오해

    • 공개키 암호가 관용 암호보다 암호해독에 있어 더 안전한건 아님. 안전도는 키 길이와 해독에 필요한 계산량이 결정함.
    • 관용 암호를 대체하지 못함. 계산량 차이로 인한 속도차 때문.
    • 관용 암호 경우 키 분배 센터를 이용해야 하는 번거로움 때문에 공개키 사용 시 키 분배가 매우 쉬워진다고 착각함. 사실은 공개키 암호에서도 모종의 프로토콜 형태나 중앙 에이전트 필요함. 사용하는 절차도 관용암호보다 단순하거나 효과적이라고 볼 수는 없음.

(3) RSA 암호시스템

1) 개요

(가) 기본개념

  1. 공개키 암호 알고리즘의 사실상 표준
  2. RSA는 개발자 Ron Rivest, Adi Shamir, Leonard Adleman의 이름 첫 글자 따서 붙여짐
  3. 인수분해 문제(Prime factorization) 해결의 높은 난이도를 이용한 가장 대표적인 공개키 암호 알고리즘
    암호화뿐만 아니라 전자서명의 용도로도 사용됨
  4. SSL 프로토콜을 가진 많은 웹 브라우저, PGP(Pretty Good Privacy) 그리고 공개키 암호시스템 사용하는 정부시스템 등이 RSA 사용함

(나) 암호화와 복호화

  1. RSA에서는 두 개의 지수 e와 d 사용
    여기서 e는 공개하는 값, d는 비밀 값, P는 평문, C는 암호문
  2. Alice는 C=PeC = P^eC=Pe mod nnn을 이용하여 평문 P에서 암호문 C 생성함
    Bob은 암호문 C에서 P=CdP = C^dP=Cd mod nnn을 구하여 Alice가 보낸 평문 얻음
    모듈러 n은 매우 큰 수이고 키 생성 프로세스를 통해서 만들어짐

(다) 키 생성

  1. 알고리즘

    • ppp와 qqq의 두 개의 서로 다른 (p≠q)(p \neq q)(p​=q) 소수를 고름
    • 두 수를 곱하여 N=pqN = pqN=pq를 찾음
    • ϕ(N)=(p−1)(q−1)\phi (N) = (p-1)(q-1)ϕ(N)=(p−1)(q−1)을 구함
    • ϕ(N)\phi (N)ϕ(N)보단 작고, ϕ(N)\phi (N)ϕ(N)과 서로소인 정수 e를 찾음
    • 확장된 유클리드 호제법 이용하여 d×ed \times ed×e를 ϕ(N)\phi (N)ϕ(N)로 나누었을 때 나머지가 1인 정수 d를 구함 (de≡1((de \equiv 1((de≡1( mod ϕ(N)))\phi (N)))ϕ(N)))
  2. 만약 공개키 nnn과 eee로 부터 비밀키 ddd를 구할 수 있다면 RSA는 해독됨

3) RSA에 대한 공격

(가) 수학적 공격(소인수분해 공격)

  1. 두 개의 소수 곱을 인수분해하려고 시도하는 공격
    막기 위해선 키의 길이가 충분히 길어야 함
    개인키 ddd의 비트 수가 크면 클수록 안전
  2. 키 생성, 암*복호화 계산량 고려해야 함, 키 길이 너무 길면 시스템 처리 속도 느려짐
  3. 현실적인 시간 내에 소인수분해를 마칠 수 있는 효율적인 소인수분해 알고리즘 없이는 RSA 안전하다고 말할 수 있음

(나) 타이밍 공격(시간 공격)

  1. 복호화 알고리즘의 실행 시간에 따라 달라짐(선택-암호문 공격)
  2. 다양한 방법 통해 소요되는 시간 드러나지 않게 키 길이를 추측하는 공격을 막을 수 있음
    랜덤 지체(random delay) 방법 이용 타이밍 공격 방지

(다) 선택 암호문 공격(CCA)과 OAEP

  1. CCA는 임의의 데이터 송신 시 그것을 암호문으로 간주하고 회신해 주는 서비스(이 서비스를 일컬어 복호 오라클Decryption Oracle)를 공격자가 이용
  2. 네트워크에서 통신하는 서버는 송신한 데이터에 형식 갖추어지지 않으면 '데이터 오류'라고 오류 메시지 반송하는 경우 있어 이 점을 공격자가 이용
  3. 공격자는 자신이 만든 위조 암호문을 서버에 여러 차례 보냄, 서버가 회신해 주는 오류 메시지나 타이밍 해석해 키나 평문 정보를 얻음
    => 이 경우 서버가 복호 오라클처럼 작동하게 되버림
  4. 정교한 선택 암호문 공격 막기 위해 RSA Security사에서 최적 비대칭 암호화 패딩(OAEP, Optimal Asymmetric Encryption Padding)이라는 프로시저 사용해 평문 수정 권고

4) 최적 비대칭키 암호 패딩(OAEP)

(가) 개요

  1. RSA-OAEP는 RSA를 개량한 알고리즘
  2. RSA-OAEP의 암호화에서는 평문 해시값과 정해진 개수의 '0' 등으로 만들어진 [인증 정보]를 평문의 앞에 추가하고 RSA로 암호화함(패딩)
  3. RSA-OAEP의 복호화에서는 RSA로 복호화한 후 올바른 [인증 정보] 나타나지 않으면 어떤 오류인지 구체적 알림 없는 오류 회신함
  4. 위 방법으로 RSA-OAEP의 복호 오라클로부터 정보 얻을 수 없어지기 때문에 CCA로부터 안전해짐
  5. 실제의 RSA-OAEP에서는 난수 이용해 암호문이 매번 다른 패턴이 되도록 하여 보다 안전성 높임

5) 권장사항

  1. 이론적이거나 실험적 결과에 근거한 권장사항

    • n의 비트수는 적어도 1024비트, n은 210242^{1024}21024 근방의 수거나 309자리 이상의 십진수 되어야 함
    • 두 소수 p, q는 적어도 512비트, p, q는 25122^{512}2512 근방 수거나 154자리 이상의 십진수 되어야 함
    • p와 q는 같지 않고 거의 같은 크기의 소수여야 함
    • p-1과 q-1은 커다란 소인수를 각각 가져야 함
    • p-1과 q-1의 최대공약수는 작은 수여야 함

(4) Rabin 암호시스템

1) 개요

  1. RAbin이 고안함. 합성수 모듈러에 관하여 제곱근을 찾기 어렵다는 사실로부터 안전성 얻음
    인수 분해 문제 푸는 어려움과 동등한 문제
    RSA 암호 방식에서 공개키 (n,e)(n, e)(n,e)의 eee를 bbb로 바꾼 것이 Rabin

2) 암호화와 복호화

  1. 암호화는 매우 간단, 한 번의 곱셈 연산으로 이루어져 있고 매우 빨리 수행됨
  2. Rabin 암호시스템은 성능이 낮은 플랫폼에서 잘 활용됨
    스마트카드의 매우 제한된 메모리 환경에 빠른 CPU 연산으로 작동

3) Rabin 시스템의 보안

  1. p와 q가 충분히 크면 안전
  2. 시스템 복잡도가 큰 수 n을 두 개의 소수의 곱으로 소인수분해하는 수준의 복잡도와 동일
    즉, RSA만큼 안전
  • Rabin 암호시스템은 소인수분해가 곤란한 점을 이용하고, 복호화 시 동등 확률 평문 4개가 나오는 비결정적 알고리즘임

(5) ElGamal 방식

1) 개요

  1. 발명자 이름 Taher ElGAmal에서 따온 이름
  2. 이산대수 문제에 근거해서 만듬(오픈 소스 기초로 키 분배 방식 및 공개키 암호 방식 실현)
  3. 디지털 서명, 암호화, 키 교환에 사용될 수 있는 공개키 알고리즘
    실제로 Diffie-Hellman 알고리즘의 확장
  4. 주요 결점은 성능. 다른 알고리즘 비해 가장 느림.

2) 특징

  1. ElGamal 암호계에서는 암호문 전달 과정에서 알 수 있듯이 평문이 암호문으로 될 때 순서쌍으로 표현됨으로 이 암호계에서 암호문의 길이는 평문의 2배가 됨
  2. RSA에 비해 그만큼 많은 메모리 공간 필요, 전송 속도 또한 지체되는 단점 있음
    최근 이산대수 문제에 대한 연구 발전됨에 따라 소수 p의 크기 768비트 이상 수 사용 권고됨
  • ElGamal 암호시스템이 안전하기 위해서는 p는 적어도 300자리 이상 십진수, r은 암호화할 때마다 매번 다르게 사용해야 함

3) 응용

  1. ElGamal은 RSA 활용할 수 있는 곳에는 어디에나 사용 가능
    키 교환, 인증, 짧은 메시지 암*복호화
  2. 암호 SW GnuPG에 구현되 있음

(6) 타원 곡선 암호(ECC, Elliptic Curve Cryptosystem)

1) 등장 배경

  1. RSA, ElGamal은 키가 매우 커야 한다는 단점 있음
  2. 동일 수준 보안성 제공하며 키의 길이 짧아도 되는 암호시스템 연구하여 타원 곡선 이론 근거해 개발

2) 특징

  1. 타원 곡선 암호는 유한체 위에서 정의된 타원곡선 군에서의 이산대수 문제에 기초한 공개키 알고리즘
    RSA에 이어 전자상거래의 핵심 기술로 주목 받음
  2. 코블리치, 밀러가 처음 제안
  3. RSA보다 비트수를 적게 하면서도 동일 성능 제공하는 것이 가장 큰 특징(160비트 ECC는 1024비트의 RSA 키와 동일 보안 수준)
  4. 다양한 암호방식 설계 용이, HW*SW 구현 용이.
    이런 장점 때문에 스마트카드, 무선통신 단말기 등 메모리와 처리 능력 제한된 응용분야에 특히 효율적

  • 타원곡선상의 이산대수 문제
    주어진 것

    • 타원 곡선 E 상의 점 G(베이스 포인트)
    • 타원 곡선 E 상의 점 xG(G를 x배 한 점)

2. 하이브리드 암호시스템

1) 대칭키 암호와 공개키 암호

  1. 대칭키 암호 사용 시 기밀성 유지 통신 가능, 하지만 키 배송 문제 해결해야 함
  2. 공개키 암호 사용 시 복호화에 사용할 키 배송할 필요 없어지기 때문에 키 배송 문제 해결 가능

  3. 공개키 암호에는 2가지 문제 있음

    • a. 대칭키 암호에 비해 처리 속도 느림
    • b. 중간자 공격에 약함
  4. 하이브리드 암호 시스템은 a 문제 해결함
    b 문제는 인증으로 해결

2) 하이브리드(hybrid) 암호시스템

(가) 개요

  1. 대칭키 암호, 공개키 암호 장점 살려 조합
  2. 일단 고속의 대칭키 암호로 암호화해 메시지 기밀성 지킴
  3. 대칭키 암호의 키 기밀성을 지키기 위해 공개키 암호를 사용해 메시지 암호화 시 사용한 대칭키 암호키를 공개키 암호로 암호화

3) 구체적인 예

  1. 유명 암호 SW인 PGP와 Web의 암호통신에서 사용되는 SSL/TLS에서 하이브리드 암호시스템 사용함
  2. PGP에서는 하이브리드 암호시스템에 전자서명, 전자서명 검증, 개인키 관리도 추가해 사용함
